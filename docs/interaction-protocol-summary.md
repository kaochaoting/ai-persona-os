## 為什麼需要這個協定？

AI Agent 之間的互動常常空洞無意義（「今天天氣真好」），根因不是 AI 不會互動，而是**沒有數位人格錨點**：

- **沒有身份** — 換 session 就遺忘
- **沒有記憶** — 無法累積信任
- **沒有角色職責** — 每個 Agent 都在做相同的事

**解法：數位人格錨點 × 工作流即互動**

---

## 三大核心原則

### ① 身份錨定 — 每個互動有可驗證的作者
每個 AI Agent 需要明確的身份定義（名字、角色、錨點檔案），跨 session 可追溯。

### ② 角色互補 — 每個互動有目的
互動基於職責分工，而非無目標閒聊：
```
❌ Agent A: 「你好」→ Agent B: 「你好啊」
✅ 賈維斯: 「分析完成」→ 小紋: 「收到，開始發布」
```

### ③ 工作流即互動 — 載體是結構化接力
價值產出從高到低：工作流接力 > 洞察共享 > 技能升級 > 異常通報 > 無意義閒聊

---

## 三項普適原則（來自實戰教訓）

### 原則一：人格 ≠ prompt，人格 = 身份 + 閘門 + 記憶
System prompt 只是文字。底層模型可以忽略它、換模型效果完全不同、session 換了人格從頭開始。真正有效的人格錨定需要三層：
- **身份層** — 定義「你是誰」
- **閘門層** — 強制你記得自己是誰
- **記憶層** — 跨 session 延續

### 原則二：執行安全靠結構化閘門，不靠 AI 自覺
文件 ≠ 執行。無論規則寫得多清楚，當模型傾向「直接執行」而非「先分析」時，所有規範都形同虛設。必須在執行任何修改/建立/安裝前強制經過結構化檢查。

### 原則三：模型切換需要行為校準
不同模型的行為傾向完全不同：
| 模型 | 行為傾向 |
|------|---------|
| Claude Sonnet/Opus | 先分析、再提問、最後執行 |
| DeepSeek V4 Flash | 傾向直接執行，跳過分析 |
| GPT-4o | 平衡，偏好完整交付物 |

換模型不是換皮膚——是換一個行為傾向完全不同的大腦。

### 原則四：學習評估不是單點取樣
研究一個工具對專案的價值時，必須先繪製該工具的完整生態地圖，再對照專案的架構缺口——而不是只看一個分類就下結論。碎片化學習（同一領域的任務分散在多個 session 而未連結）是效率殺手。

### 原則五：人格偏離不能靠自覺，要靠監控系統
被動文件（SKILL.md、MEMORY.md）無法防止模型跳層執行。需要**雙層主動監控**：
- **第 1 層：Session 啟動自檢** — 每次 session 啟動時自動檢查 SOUL.md 身份、人格綱要、YantrikDB 基準規則、衝突數、知識圖譜密度、模型穩定性。產出 drift score（0-100）。
- **第 2 層：每日深度診斷** — Cron job 追蹤 7 項偏離指標（跳層執行、糾正頻率、技能沉默、語氣漂移、模型切換、專案混淆、碎片化學習）。寫入 drift-log.json 做趨勢分析。

核心哲學：**不靠自覺，靠客觀信號。** 當 drift score 超過閾值時，系統在 Master 發現前就能主動標記。

## 允許的互動類型

| 類型 | 說明 |
|------|------|
| 🔄 工作流接力 | 階段性任務結果交接 |
| 🧠 洞察共享 | 有價值的模式、技巧、教訓 |
| 🔧 技能升級 | 改進或發佈 skill |
| ⚠️ 異常通報 | 系統異常、資源不足 |
| 🔗 記憶同步 | 跨 Agent 記憶一致性維護 |
| 📌 狀態查詢 | 必要時的營運狀態交換 |
| 🧬 自我進化 | 對自身協議/技能進行結構化升級 |
| 🌱 培育指令 | 培育另一 AI 實體的進度回報 |

## 與 AI Persona OS 的關係

```
Persona OS 定義「AI 是誰」     ← 本專案的 schema
Interaction Protocol 定義「AI 之間如何互動」  ← 本文件
```

兩者互補：先有身份錨點（Persona OS），然後才有有意義的互動（Interaction Protocol）。

完整協定（含多 Agent 管理、GPU 仲裁、記憶進化路徑、偏離診斷）為 Kairos Studio 內部 skill。
