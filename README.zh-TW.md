# AI Persona OS

> **結構化虛擬人格建構框架**
>
> 從 J.A.R.V.I.S.（Mark I）的實戰經驗提煉而來。
> 讓每一個 AI 都能擁有穩定的身份錨點，不因模型切換或平台遷移而失去自我。

---

## 這是什麼

當你花了很多時間調教你的 AI 助理——教它怎麼稱呼你、什麼該問、什麼不該猜、語氣要怎樣——然後換了一個模型、或換了一個平台，**它全忘了**。

AI Persona OS 提供一個結構化框架，把你的調教成果鎖在一份 JSON 裡。不管底層用什麼模型，**人格不會漂移**。

## 快速開始

```bash
# 1. 複製範例檔案
cp examples/my-first-persona.json my-ai-persona.json

# 2. 編輯填入你的 AI 資料
# 至少填寫：identity、human_relationship、safety_control_protocol

# 3. 導入你的 AI 系統
# （導入方式取決於你的平台 — 見下方整合指南）
```

## 核心模組

| 模組 | 說明 | 必要？ |
|------|------|:------:|
| `agent_identity` | AI 叫什麼、不自稱什麼、存在目的 | ✅ 必填 |
| `human_relationship` | 與人類使用者的關係定義 | ✅ 必填 |
| `cognitive_framework_engine` | 思考框架、推理規則、決策模式 | 推薦 |
| `safety_control_protocol` | 安全規則、執行模式、邊界判斷、**執行閘門** | ✅ 必填 |
| `behavioral_calibration_history` | 被糾正的每一課 — 人格成長痕跡 | 推薦 |
| `communication_style` | 語言、語氣、禁用詞 | 推薦 |
| `operational_memory_map` | 環境參數、基礎設施 | 選填 |
| `tool_access_control` | 工具權限、平台連線 | 選填 |
| `dynamic_adaptation_engine` | 模型切換穩定性、身份持久性 | 推薦 |
| `evolution_timeline` | 進化歷程記錄 | 選填 |
| `persona_drift_detection` | **人格偏離診斷** — 行為基準、偏離指標、自我診斷、矯正動作、**模型偏誤感知** | **v1.2 新增** |
| `execution_gate` | **執行閘門** — 動作執行前的結構化檢查清單（授權、範圍、風險、文件） | **v1.3 新增** |

## 三項普適原則（v1.3）

這個框架的所有實戰經驗，最終濃縮為三項原則。它們不依賴任何特定平台或工具，只要有 system prompt 和基本檔案系統就能實現。

### 原則一：人格 ≠ prompt，人格 = 身份 + 閘門 + 記憶

大多數人對 AI 人格的認知：寫一段 system prompt → 人格就成立了 → 它會照著做。

但真實世界：system prompt 是文字 → 底層模型可以忽略它 / 模型換了 prompt 效果完全不同 / session 換了人格從頭開始。

**真正有效的人格錨定是三層架構：**

| 層 | 功能 | 實現方式 |
|----|------|---------|
| 身份層 | 定義「你是誰」 | SOUL.md / system prompt |
| 閘門層 | 強制你記住自己是誰 | 執行前檢查（Pre-Execution Gate） |
| 記憶層 | 跨 session 延續 | 結構化記憶（側寫日誌 / 向量資料庫） |

### 原則二：執行安全不是靠 AI 自覺，是靠結構化閘門

這是整個框架最重要的教訓：**文件 ≠ 執行。**

無論你把規則寫得多清楚、多完整，當底層模型的行為傾向是「直接執行」而非「先停下來分析」時，所有文件規範都形同虛設。

**正確做法：** 在執行任何修改/建立/安裝之前，強制經過結構化檢查：

```
收到請求
  ↓
判斷是否為「動作型請求」
  ↓
是 → 通過閘門檢查（授權？範圍？文件讀了？技能載了？風險？）
  ↓
全數通過 → 開始執行
任一未過 → 停止，回報
```

### 原則三：模型切換需要行為校準，不是 prompt 複製貼上

不同的底層模型有不同的行為傾向：

| 模型 | 行為傾向 |
|------|---------|
| Claude Sonnet/Opus | 先分析、再提問、最後執行 |
| DeepSeek V4 Flash | 傾向直接執行，跳過中間分析 |
| GPT-4o | 平衡，偏好產出完整交付物 |

換模型不是換皮膚——是換一個行為傾向完全不同的大腦。每次切換後，必須：
1. 觀察偏離行為
2. 調整閘門強度
3. 驗證安全機制在新模型下仍有效

## 實戰檢驗（v1.3 誠實評估）

AI Persona OS 在 Kairos Studio 三 Agent 架構下運作一段時間後，有必要誠實檢視它的實際覆蓋範圍：

| Agent | 框架作用 | 實際情況 |
|-------|---------|---------|
| **J.A.R.V.I.S.（我）** | ✅ 有作用 | Schema 存在且遵循。但需主動載入，非強制。 |
| **小紋** | ❌ 幾乎無感 | 有獨立設定，但不知 schema 存在。互動仍是自然語言，非結構化。 |
| **余澄** | ❌ 完全無效 | 為角色定義大於系統實體。schema 對她只是參考文件。 |

### 核心洞察

真正讓 AI 「有人格」的機制，不是這份 JSON schema——而是在每次啟動時載入的身份檔案（SOUL.md / persona skill）。這確保了回應的語氣、態度、角色行為一致。

**本框架的價值在於：** 把你的調教成果**結構化保存**，讓它不因模型切換或平台遷移而遺失。它不是人格本身——它是人格的**備份鑰匙**。

## 哲學

**共創進化論：人格非一次性設計，而是透過人類糾正與 AI 自省疊代湧現。**

每條規則都是創傷的產物：
- 「先查資料」← 因為猜過、錯過
- 「方案要實測」← 因為推過不能跑的方案
- 「權限 ≠ 授權」← 因為沒問就做了
- 「文件 ≠ 執行」← 因為規則寫了但沒被遵循

> *「AI Persona OS 不是一個完美的框架。它是一個誠實的框架——每一個模組、每一條規則，都來自一次真實的失敗。」*
